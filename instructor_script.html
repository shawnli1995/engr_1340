Introduction (5 Minutes)

"Good morning, everyone. Over the next two classes, we're going to build your first complete website from scratch. We're starting with a Personal Portfolio. The reason we're doing this *before* Project 2 is that it's a more motivating and simple way to learn the core skills. You'll build something about *you*."

"The most important concept I want you to learn today is the 'Skeleton and Skin' analogy."

  • HTML (HyperText Markup Language) is the SKELETON. It's the structure, the bones of the website. It's the <h1> heading, the <p> paragraph, the <img> image.
  • CSS (Cascading Style Sheets) is the SKIN. It's the style, the paint, the clothes. It's the color: blue, the font-family, the border-radius.

Module 1: The "Skeleton" (HTML Content)

Goal: Explain the "what" and "why" of each core HTML tag.

Step 1-3: Folders & Files

Why do we do this?

We *always* start by making a single folder for our project. This keeps all our files—HTML, CSS, images—in one organized place. By opening the *folder* in VS Code, the sidebar will show our entire project structure, which makes it easy to navigate.

Why index.html? This is a special, 'magic' filename. When a web server (like GitHub Pages) looks at a folder, it *automatically* looks for a file named index.html to show as the homepage. All your homepages must be named this.

Step 4: The "Boilerplate"

What is this code?

In your empty index.html file, typing ! and hitting Tab is an 'Emmet' shortcut that gives us the basic HTML skeleton. Let's look at what it gave us:

  • <!DOCTYPE html>: This is a command. It tells the browser, "Hey, I'm an HTML5 document. Treat me as a modern webpage."
  • <html lang="en">: This is the root tag. Everything else lives inside this. lang="en" tells the browser this page is in English, which is good for accessibility and search engines.
  • <head>: This is the 'brain' of your website. You don't *see* the information in the <head>, but it's critical metadata.
  • <meta charset="UTF-8">: This tag sets the character encoding. It's what allows your webpage to use any character, symbol, or emoji from any language in the world.
  • <meta name="viewport" ...>: This is one of the most important tags for the modern web. It tells the phone, "Make the webpage as wide as your screen," and "Don't zoom in." This is what makes your website look good on mobile.

Step 5: Change Your Title

Why <title>?

This is the text that appears on the browser tab. It's also what Google uses as the main blue link in its search results. It's very important for Search Engine Optimization (SEO).

Step 6: Add Your Main Heading

Why <h1>?

This is a Heading 1. It's the most important heading on your page. You should only have one <h1> per page. It's the title, and search engines pay a lot of attention to it.

Step 7: Add a Subheading

Why <h2>?

This is a Heading 2, or a sub-heading. It's for section titles. You can have as many of these as you want. They create the document's outline. Notice it's slightly smaller than the <h1>.

Step 8: Add a Paragraph

Why <p>?

This is a Paragraph tag. This is the default tag for all your main text blocks. The browser automatically adds a bit of space (margin) before and after each paragraph.

Step 9: Add a List

Why <ul> and <li>?
  • <ul>: This is an Unordered List tag. It doesn't mean a list in a random order; it just means a *bulleted* list. This tag is the *container* for the list items.
  • <li>: This is a List Item. Each <li> is one bullet point. You can't have an <li> by itself; it must always be inside a <ul> or an <ol> (an Ordered, or numbered, List).

Checkpoint 1: "Look at your page. It's just black text on a white background. It's structured, but it's ugly. This is raw, unstyled HTML. We've successfully built the skeleton. Now, let's add the skin."

Module 2: The "Skin" (Basic CSS)

Goal: Explain CSS selectors, properties, and the "box model".

Step 1-2: Create CSS Files

Why a separate folder?

Organization. In a real project, you might have many stylesheets. We keep them all in a css folder. This is a best practice called "separation of concerns."

Step 3: Link Your CSS

Why <link>?

This is the bridge between your HTML and CSS. If this one line is wrong, *none* of your styles will show up. This is the #1 place students make a mistake.

  • rel="stylesheet": rel stands for 'relationship.' This tells the browser *what* this file is.
  • href="css/styles.css": href is the 'hyperlink reference.' This is the *path* to the file. It tells the browser, "Look in the css folder, and find the file named styles.css."

Step 4: Test the Link

What is a CSS Rule?

body { ... } is your first CSS Rule. It has two parts:

  1. body is the Selector. It says *who* we want to style.
  2. { ... } The curly braces hold the Declarations (the styles).

background-color: #f0f4f8; is a Declaration. background-color is the Property (what to change), and #f0f4f8 is the Value (what to change it to). This is a 'hex code' for a color—a light gray.

Step 5: Improve the Font

Why these properties?
  • font-family: Arial, sans-serif;: This is a 'font stack.' The browser tries to find Arial. If it can't, it finds the next one. sans-serif is the generic fallback.
  • line-height: 1.6;: This controls the spacing *between* lines of text. 1.6 means "1.6 times the font size." It makes text much easier to read.

Step 6: Center Your Content

Why <div class="container">?
  • <div>: A <div> is the most basic, generic box in HTML. It doesn't *mean* anything—it's just a "division" or "container" we can use to group other elements.
  • class="container": A class is a *nickname*. We are giving this <div> the nickname "container." This lets us style *just this box* and any other element we give the same nickname.

Step 7: Style the Container

Why .container? (The "Box Model")

In CSS, a period . selects a class. So this rule means "Find any element with the nickname 'container' and apply these styles."

  • width: 800px;: We're setting a fixed width. This stops our content from stretching to the edges of a wide monitor.
  • margin: 0 auto;: This is a classic centering trick. 'Margin' is the space *outside* an element. This rule means "0 margin on the top and bottom, and auto margin on the left and right." The browser automatically calculates equal margins, which perfectly centers the box.
  • background-color: #ffffff;: We're making the card white, so it stands out from our gray page background.
  • padding: 20px;: 'Padding' is the space *inside* the box, between the content and the edge of the box. Margin is outside, Padding is inside.
  • border-radius: 8px;: This just rounds the corners of our box.
  • box-shadow: ...;: This adds a subtle shadow to give the card a 3D, 'floating' effect.

Step 8: Style Your Headings

Why Element Selectors?

h1 and h2 are Element Selectors. They find *every* <h1> and <h2> tag on the page and apply these styles. This is great for making all your headings consistent.

Module 3: Media & Better Structure

Goal: Explain semantic HTML and how to add media (images and links).

Step 1-2: Add Images

Why an images folder?

Same as our css folder: organization. All our images live here.

Step 3: Refactor Your HTML

Why Refactor?

We are about to refactor our code. This is a key programming term. It means we are *changing the structure* of our code to make it better, cleaner, or more efficient, *without* changing what it does.

We're replacing generic <div>s with Semantic Tags—tags that have *meaning*:

  • <header>: This tag tells the browser and screen readers, "This is the top banner of the page."
  • <main>: This tag says, "This is the *main, unique* content for this page."
  • <footer>: This tag says, "This is the footer, the bottom of the page."

Why do this? 1) **Accessibility:** A screen reader can now say, "Now jumping to the main content." 2) **SEO:** Google understands your page structure better. 3) **Clarity:** It's much easier for *us* to read our own code.

Step 4: Add New CSS

Why new selectors?
  • header { text-align: center; }: We're telling the <header> element to center all its contents.
  • header img { ... }: This is a Descendant Selector. It means "find any <img> tag that is *inside* a <header> tag." This is powerful because it *won't* style other images on our page.
  • border-radius: 50%;: This is a great trick. If you have a perfect square (width: 150px and height: 150px), a border-radius of 50% will make it a perfect *circle*.
  • <a>: This is the Anchor tag. It's what makes the web a *web*. It creates hyperlinks. href is the *destination* of the link.
  • footer a { ... }: Another descendant selector. "Style any <a> tag that's inside a <footer>."
  • footer a:hover { ... }: This is a pseudo-class. It styles the link *only* when the user's mouse is hovering over it. We're adding the underline back on hover, which is a great visual cue for the user.

Module 4: Publishing Your Site (GitHub Pages)

Goal: Explain the Git workflow and the concept of "hosting."

"This is the final and most exciting part. Right now, your website only lives on your computer. We need to put it on a server—a special computer that's on 24/7."

"GitHub has a free service called GitHub Pages that will be our server."

Step 1: Create a GitHub Repo

Why?

A Repository (or 'repo') is just a folder for a project. We're creating a new, *empty* project folder on GitHub's servers. We make it Public so GitHub Pages can access it.

Step 2: Clone Your Repo

Why git clone?

This command means "go to this URL, copy that entire repository, and download it to my computer." We do this first to get a folder that is *already connected* to the cloud.

Step 3: Move Your Files

Why?

We are moving our local-only files (our index.html, css, and images) into the new, GitHub-connected folder.

Step 4: Upload Your Files

Why? (The 3-Step-Git-Dance)

This is the most important workflow for this class. You will do this *all the time*. It's three steps:

  1. git add . (Stage)
    Think of this as putting your files into a cardboard box. You're *preparing* them to be saved. The . just means "add everything in this folder."
  2. git commit -m "..." (Commit)
    This is the "save" step. You're sealing the box and writing a *message* on it (that's the -m). The message says *what* you did, like "Added portfolio website files."
  3. git push (Push)
    This is the "upload" step. You are 'pushing' your 'commit' (the sealed box) from your computer up to the GitHub server.

Step 5: Activate GitHub Pages

Why?

Our code is on GitHub, but it's not a *website* yet. We just have to flip the switch.

When you go to Settings > Pages and select main and / (root), you are telling GitHub: "Hey, see this main branch? I want you to treat it as a live website. Look for an index.html file in the root folder and *serve* it to the public."

Step 6: You're Live!

Why?

It takes about 1-2 minutes for GitHub's servers to build and deploy your site. Once it's done, that URL is live on the internet. You've successfully built and deployed a real website!
